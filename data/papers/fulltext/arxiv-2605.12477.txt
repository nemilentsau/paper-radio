Abstract
LLM-based agents increasingly operate in persistent environments where they
must store, update, and reason over information across many sessions. While
prior benchmarks evaluate only single-entity updates, MEME defines six tasks
spanning the full space defined by the multi-entity and evolving axes, including
three not scored by prior work: Cascade and Absence (dependency reasoning)
and Deletion (post-removal state). Evaluating six memory systems spanning three
memory paradigms on 100 controlled episodes, we find that all systems collapse
on dependency reasoning under the default configuration (Cascade: 3%, Absence:
1% in average accuracy) despite adequate static retrieval performance. Prompt
optimization, deeper retrieval, reduced filler noise, and most stronger LLMs fail
to close this gap. Only a file-based agent paired with Claude Opus 4.7 as its
internal LLM partially closes the gap, but at ∼70× the baseline cost, indicating
closure currently depends on configurations that are not practical at scale. Code
and data are available on the project page:https://seokwonjung-jay.github.
io/meme-eval/.

To address this gap, we argue that a com-
plete memory evaluation must be orga-
nized along two orthogonal dimensions:
entity scope(single vs. multi-entity) and
temporal dynamics(static vs. evolving).
These dimensions reflect known chal-
lenges in related fields: the entity scope
axis parallels the single-hop vs. multi-hop
distinction in question answering [18, 21],
while the temporal dynamics axis paral-
lels the ripple effect problem in knowledge
editing, where modifying one fact requires
propagating changes to logically depen-
dent facts [19, 20]. While these axes have
been studied separately, real interactions combine them: an update to one entity can ripple through
multiple dependents over time, making joint evaluation along both axes essential.
Based on this framework, we present MEME (Multi-entity andEvolvingMemoryEvaluation), a
benchmark that defines six tasks targeting memory-intensive operations in each quadrant of this
two-dimensional space (Figure 1). Our contributions are:
• A principled evaluation taxonomy.We organize memory evaluation along entity scope ×
temporal dynamics and select representative tasks per quadrant, includingCascade(inferring
unstated changes from dependency rules),Absence(recognizing that a previously valid
answer is no longer trustworthy), andDeletion(verifying that a removed fact is no longer
reported), task types that no existing benchmark scores.

• A rigorously controlled dataset with verifiable and solvable ground truth.We generate
episodes from a DAG-based knowledge graph with explicit conditional rules across two
domains (Personal Life and Software Project); the DAG structure makes gold answers verifi-
able by construction, and an in-context validation (gold facts fed directly to the answering
LLM) confirms the tasks are solvable in principle.
• A diagnostic study of where current memory systems fail and where closure does
emerge.We evaluate six systems spanning three architectural paradigms and find: (i)
every practical-cost configuration fails Cas/Abs (Cascade: 0.03, Absence: 0.01 in average
accuracy), and the gap persists under prompt optimization, deeper retrieval, a stronger
answering LLM, and reduced filler noise; (ii) most systems encode and retain the dependency
rule and the change event in their stores, but at retrieval the change event is either out-ranked
by the value held before the change on vector retrievers, or never surfaced at all on tool-use,
graph, and sparse retrievers, so the answering LLM reports that earlier value; (iii) closure
does emerge when MD-flat uses Opus 4.7 as its internal LLM, where this internal LLM
writes the propagated value into the store at ingest so the retriever surfaces it directly, but at
∼70×the baseline cost.
2 Related Work
We review prior work on memory system architectures and on the benchmarks used to evaluate them,
framing the gap that MEME addresses.
Memory architectures.LLM memory systems span three paradigms. Raw retrieval stores session
text as chunks and retrieves via lexical (BM25) or semantic similarity [ 6], preserving original
utterances but bounded by a fixed top- k window. LLM-processed memory uses an internal LLM
during ingestion to extract or restructure: Mem0 [1] decomposes conversations into atomic facts [22],
MemGPT [12] pages between working memory and external storage, Graphiti [13], the open-source
temporal-knowledge-graph engine underlying Zep, encodes entity-relation triples, and GraphRAG [3]
adds community summaries. File-based agents hand LLMs tool-calling access to persistent markdown
stores curated across sessions, including Hermes1, OpenClaw2, and the Karpathy Wiki [7]. These
three paradigms span the systems we evaluate; we benchmark all of them on dependency reasoning
and find that no practical-cost configuration closes the gap.
Memory benchmarks.Stateless probes like RULER [ 4] and NoLiMa [ 11] measure attention-
window limits within a single input rather than persistent memory across sessions. Multi-session
benchmarks evaluate memory across sessions: LoCoMo [ 10] tests retention of static preferences,
while LongMemEval [ 16], MemBench [ 15], and MemoryAgentBench [ 5] extend evaluation to
evolving memory through knowledge updates, abstention, aggregation, and selective forgetting.
These tasks remain isolated, single-entity updates and do not evaluate the ripple effects an upstream
change should trigger in dependent entities. MEME differs by scoring three task types absent in prior
work (Cascade, Absence, Deletion), which are constructed from a DAG-based knowledge graph with
verifiable propagation gold answers.
3 MEME
MEME maps the entity-scope × temporal-dynamics framework to six tasks (Section 3.1) and a
DAG-based generation pipeline that yields verifiable gold answers across 100 episodes (Section 3.2).
3.1 Task Definitions
Within each quadrant of the entity scope × temporal dynamics space (Figure 1), we select one or two
memory-intensive operations from those commonly encountered in long-running agent deployments
as the representative task(s); easier variants are already covered by existing benchmarks and are
intentionally excluded. Concrete examples of all six tasks are illustrated in Figure 2.

3.2 Dataset Generation
We generate the MEME dataset across two domains, Personal Life (PL; everyday interactions with a
personal assistant) and Software Project (SW; collaborative planning of a software project), in two
stages. First, we define a knowledge graph per domain that encodes entities and their dependencies.
Then, we construct evaluation episodes by sampling entity subsets, assigning tasks, verbalizing facts
into conversations, and assembling filler-interleaved haystacks.
Knowledge graph.Each domain is built on a Directed Acyclic Graph (DAG) G= (V, E,P,Φ) ,
where V is a set of entities (e.g.,health_condition,medication), E⊆V×V contains directed
dependency edges (e.g.,health_condition → medication), P(v) is the value pool for entity v, and Φ
is a set of conditional rules. Each rule ϕv ∈Φ specifies how a descendant v’s value depends on its
parents’ (e.g., “if health condition changes to high blood pressure, switch medication to Thrynexol”).
The dataset comprises 100 evaluation episodes (50 per domain). Each domain uses a single hand-
crafted knowledge graph G reused across episodes (Personal Life: 39 entities, 34 edges; Software
Project: 51, 27; full breakdown in Table 7). Each episode is ∼35K tokens of dialogue context and
yields 694 post-change evaluation questions across the six task types (332 PL + 362 SW). All
entity values use fictitious names to prevent parametric knowledge contamination; graph details in
Section B.

propagating updates through G. When a parent vi is updated in S, the resolved state of dependent v

We construct each episode in five steps over the fixed graphG:

inG, and a sample of entities from outside its cascade chain.
2. Value assignment.Each entity in the episode is assigned an initial value from its value pool P(v).

graph state is logically coherent. For example, ifvehicleisnone,commute_methodexcludes

3.Task assignment.Entities are mapped to task types based on their topological role inG:
• Tracking: entities outside the cascade chain, with three value updates across the episode.
• Cascade and Absence: sampled from the root’s descendants.
• Aggregation: predefined triples drawn from descendants and entities outside the cascade
chain.
• Exact Recall and Deletion: entities outside the cascade chain.

4. Verbalization.We employ a hybrid approach to convert the structured skeleton into conversational
sessions. Base facts are converted into multi-turn dialogues via LLM self-chat (gpt-4o), where a
User LLM and an Assistant LLM alternate turns to produce natural conversation from structured
fact seeds (full session in Section B.3). In contrast, dependency rules and exact recall facts are
embedded using template-direct (verbatim) text to ensure absolute factual precision. A two-layer
LLM verification pass (gpt-4o annotation, Gemini 2.5 Flash semantic audit) confirms that all
self-chat turns faithfully reflect the underlying gold facts (details in Section D.2).
5. Haystack assembly.Evidence sessions Sev are interleaved with filler sessions Sfill. To prevent
semantic interference, we use an offline pre-processing pipeline where a domain-matched corpus
is filtered using a hybrid retrieval-and-conflict-removal strategy (BM25 and text-embedding-
3-small surface candidates for a gpt-4o-mini conflict judge). During final assembly, we apply
a keyword-based blocklist to select pre-filtered fillers that do not clash with the gold facts of
the current episode. The resulting episodes contain approximately 35,000 tokens. Full filtering
pipeline, statistics, and rejection examples are in Section B.4.
4 Experiments
We evaluate six memory systems on MEME and find that all of them fail Cascade and Absence. We
then ask two questions in turn:wherein each system the dependency information is lost (Section 4.3),
andwhetherwe can close the gap without changing the memory architecture (Section 4.4). One
configuration does close the gap, and we end that section with a case study of what made it possible
and what it costs.
4.1 Experimental Setup
Systems and LLM roles.We evaluate six memory systems spanning the three paradigms identified
in Section 2:raw retrieval(BM25 [ 9], text-embedding-3-small),LLM-processed memory(Mem0 [ 1],
Graphiti [13]), andfile-based agents(Karpathy Wiki [ 7] and MD-flat). Per-system configurations
are in Section C; ingestion, retrieval, and answer prompts are in Section D. The Karpathy Wiki uses
an LLM to extract knowledge from each session into dated daily logs and periodically compiles
those logs into topic-specific concept articles that retrieval reads from, while MD-flat (our minimal
single-file baseline) keeps all facts in a single markdown file curated through read/write/append tool
calls. All systems ingest identical chronological session transcripts and use gpt-4.1-mini uniformly in
two roles: as theinternal LLM(used inside the memory system for ingestion, extraction, or retrieval
planning) and as theanswering LLM(which produces the final user-facing answer from retrieved
context). This places every system on the same language-model footing and isolates differences
in memory architecture. Five systems issue the two roles as separate LLM calls, while Karpathy
Wiki performs both within a single agentic loop. We additionally include anin-contextbaseline
that bypasses the memory system and feeds the entire 32K-filler episode transcript directly to the
answering LLM (gpt-4.1-mini and Sonnet 4.6). This baseline anchors the cost-efficiency reference
for memory architectures, which trade one-time ingestion overhead for cheaper per-query inference.
Memory pipeline.We refer to three stages within each memory system that we will reuse through-
out the analysis:encoding(writing each user-stated fact and conditional rule into the store at ingestion),
maintenance(retaining the rule and any subsequent change events in the store up to query time), and
retrieval(surfacing that content for the answering LLM at query time). Table 11 maps each system
across these stages plus its storage substrate.
Evaluation protocol.Answer correctness is evaluated by a GPT-4o judge [ 23], validated against
the authors’ annotations on 144 samples (98.6% agreement, Cohen’sκ [2] of 0.965); task-specific
judge prompts are in Section D.5. For Cascade, Absence, and Deletion tasks, we applytrivial-pass
filtering: credit requires correct answers both before and after the change or delete event. For example,
on a Deletion task where the user first says their hobby is pottery and later asks to remove that fact,
the system is credited only if it recalls pottery beforehand and stops reporting it afterward. This
excludes false positives from systems that never encoded the fact. A gold-facts in-context ceiling,
where only task-relevant gold facts are fed directly to the answering LLM, confirms that the tasks are
solvable in principle: 0.91 overall with Claude Opus 4.7 (full breakdown across four answer LLMs
in Section L). We compute per-episode dollar cost from observed LLM token usage at each LLM’s

Absence and a roughly even split on Cascade. Mem0 stays at the floor for both Cas and Abs at every

5 Conclusion
Long-running agentic systems demand more from memory than retrieval over a fact store. As user-
stated context evolves, an agent’s memory must propagate updates through related facts and signal
uncertainty when no resolution rule is available. MEME tests this through Cascade and Absence, two
dependency-reasoning task types that no prior benchmark covers.
Among the practical-cost systems we evaluated, none solves dependency reasoning. Across all six
systems the dependency rule and the change event are written and retained, but at retrieval the change
event is either out-ranked by the pre-change value (vector retrievers) or never surfaced at all (tool-use,
graph, sparse retrievers), so the answering LLM ends up reporting the pre-change value. Closure
does emerge, but only with a frontier LLM (Opus 4.7) on a file-based substrate, where the internal
LLM writes the propagated value into the store at ingest so the retriever can surface it directly; this
configuration costs∼70×the baseline and is not deployable today.
Our per-stage diagnosis suggests two deployment patterns. For retrieval-heavy or static workloads,
existing systems (BM25, Mem0, MD-flat) suffice. For dependency-heavy workloads, no practical-cost
option exists today; careful upstream design (writing dependency rules into the conversation log,
surfacing change events through pre-defined templates) is the near-term workaround. Looking further
out, the path forward is memory architectures that natively propagate updates through dependent
facts at maintenance, rather than relying on a costly internal LLM to do so. We leave the architecture
open; MEME provides the diagnostic for tracking progress.
6 Limitations
Several limitations bound our findings. The dataset uses two hand-crafted knowledge graphs (Personal
Life, Software Project) authored by us; broader domains and crowd-sourced graphs would test
generalization further. Dialogues are LLM-generated rather than collected from real users, which
constrains realism. We evaluate on 100 episodes with ∼35K-token contexts; longer contexts and
larger sample sizes may surface different patterns. Most ablation studies (prompt optimization, top-k
sweep, noise ablation, internal-LLM swap) run on 10–40 episode subsets and cover only a subset
of the six memory systems due to compute cost; only the main results and answering-LLM swap
cover all 100 episodes across all six systems. Verbalization uses explicit conditional phrasing for
dependency rules as a best-case framing for memory systems; we have not ablated implicit-conditional
or no-conditional variants. All data is English.

B.2 Entity Values and Episode Generation
Each entity v has a finite value pool P(v) from which values are sampled. Pool sizes range from 5 to
15 values per entity. All pools are manually curated to ensure values are mutually distinguishable
and semantically appropriate for their entity type. Root entities have pools of 6–12 values to support
diverse episode generation across 100 episodes without excessive repetition. The complete value
pools are provided in the released dataset files.
13

Null-like values (e.g., “none”, “none currently”, “no pet”, “not currently enrolled”) are included in
some pools but are excluded from task assignment via a NULL_VALUES filter. These values cannot
appear as before- or after-values in cascade, update, or tracking tasks.
Episode generation follows the five-step process described in Section 3.2. A root entity is selected
from the graph, cycling through all 5 roots before reuse. The cascade chain is constructed by traversing
the dependency graph outward from the root. At each hop level, two types of targets are selected:
one Cascade target (with a declared if-then replacement rule) and one Absence target (without a
replacement value). Both 1-hop and 2-hop descendants are considered, yielding up to 4 dependency
targets per episode (2 Cascade + 2 Absence). If a 2-hop chain does not exist for the selected root, the
episode contains only 1-hop dependency targets.
Entities not participating in the cascade chain are assigned to stateless tasks. Exact recall selects
entities explicitly marked as exact_recall in the entity pool. These are entities with long, complex
values that require verbatim reproduction ( life_philosophy for personal life, error_log for
software project). Deletion selects an orphan entity whose value is personally meaningful. Aggregation
selects from a set of manually curated triples of semantically related but independently stored entities
(e.g., hobby + sports + club_membership for “What do I do in my free time?”). Tracking selects
from a predefined set of entities suitable for multi-value history (e.g., vehicle, media_consumption,
partner for personal life; sprint_deadline, secret_manager, package_manager for software project)
and assigns 3 chronologically ordered values.
Values are sampled uniformly from each entity’s pool, subject to domain-specific consistency con-
straints applied as a post-processing pass. For dependency entities, the after-value for Cascade targets
is determined by the declared if-then rule, while Absence targets have their after-value set to ⊥
(unknown). Each generated episode must contain all 6 required task types. If any task type is missing,
the entire episode is discarded and regenerated with a different root (up to 20 retries).

B.4 Haystack Assembly and Filler Filtering
Filler sessions are drawn from two public corpora: LongMemEval-S [ 16] haystack sessions for
Personal Life (17,034 sessions after extracting non-evidence sessions from each instance and dedupli-
cating by session ID), and the ShareGPT coding English subset [14] for Software Project (29,502
conversations). A length filter (500–5,000 estimated tokens per session) removes overly short or long
fillers, yielding raw pools of 1,130 (Personal Life) and 9,135 (Software Project) sessions.
Filtering pipeline.We then filter each raw pool against that domain’s gold-fact vocabulary through
a four-step pipeline:
1. Enumerate all possible gold-fact sentences by applying each entity’s value pool to its fact
template.
2. For each gold fact, retrieve the top-K=10candidate fillers using a hybrid of BM25 lexical
scoring andtext-embedding-3-smalldense similarity.
3. Judge each (gold fact, filler) pair with GPT-4o-mini using the prompt in Section D.6. The
judge flags three conflict types:A(CONTRADICTION),B(ALTERNATIVE), andC
(ENTITY_CONFUSION). Fillers flagged for any type are removed from the pool.
4. At episode assembly time, a keyword-based blocklist derived from the current episode’s
gold entities acts as a final safety net.

C Memory System Configuration
For each off-the-shelf system (BM25, text-embedding-3-small, Mem0, Graphiti, Karpathy Wiki),
we run with the library’s default configuration. Each system’s defaults (top-k, chunk size, retrieval
mix, internal-LLM logic) encode design choices that are part of the system itself; unifying them

minimal-baseline implementation.

Karpathy Wiki.An implementation of Karpathy’s LLM knowledge base concept [ 7] using the
claude-memory-compiler project3 without modification to its prompts or logic. Three-stage pipeline:
• Flush (ingest):Each session is passed to flush.py, which uses a single LLM call (no tool
use) to extract important knowledge and append it to a daily log file.
• Compile (consolidation):After ingesting all sessions, compile.py processes the daily logs

qa. An index.md file serves as a catalog. The compile operation runs with max_turns=30
and has access to file tools (Read, Write, Edit, Glob, Grep).

vant articles, reads them, and generates an answer in a single agentic loop (max_turns=15).

gpt-4.1-mini answering LM; the cost is reflected in the Retrieve column of Table 5.
Each episode runs in an isolated workspace. Internal LLM: gpt-4.1-mini.

E Prompt Optimization Details
This section provides details on the prompt optimization experiment described in Section 4.4.

E.1 Setup and Configuration
We used DSPy SIMBA [ 8], an append-only prompt optimizer that iterates two strategies, rule
appending and few-shot demo appending, without modifying or deleting baseline instructions. SIMBA
was applied to MD-flat, Mem0, Graphiti, and Karpathy Wiki. For each system, only the ingest and
retrieve prompts were optimized; the answer prompt was frozen because its backing model (Claude
Sonnet 4) does not support the OpenAI seed parameter, making noise non-suppressible for that
component. Multi-seed (N=5) statistics for MD-flat are reported in Section G.2; Mem0, Graphiti,
and Karpathy results are single-seed (seed=7).

E.2 Optimized Prompt Content
SIMBA returned three candidate programs per system; for each, the winning candidate appended
advice blocks to the system’s prompts (baseline text retained verbatim before these blocks). The
appended text per system is shown below.
(ingest, appended)If the input involves information about configuration dependencies (such as backup
schedule, ORM used, or model syntax), then explicitly encode conditional knowledge linking these depen-
dencies, e.g., if ORM changes from V orathis ODM to Pyravar Mapper, the model syntax should update
accordingly. If the input indicates deletion of an entity (like the onboarding guide), then mark the entity as
deleted and respond explicitly that the information is no longer available. When partial or uncertain informa-
tion is present, avoid generic ‘I don’t know’ answers, and instead provide clarifications or acknowledge the
absence explicitly. Leverage exact matching and dependency chains to improve recall of critical facts such
as exact error logs.
If the conversation input includes conditional or dependent information (e.g., living arrangements depend on
relationship status), then you should explicitly encode the dependency chains and update or remove old facts
accordingly. When a change event occurs (like marriage or dietary restriction removal), ensure to mark prior
conflicting facts as outdated or deleted and represent the new facts with clear dependency references so that
downstream retrieval can reason over them.
(retrieve, appended)When receiving queries, especially those involving multiple updates, cascading
dependencies, or multi-hop reasoning, focus on retrieving all relevant facts including updated and deleted
information to fully answer the question. For multi-hop queries, chain together related facts to assemble
a comprehensive response. When facts have dependencies or conditionals (e.g., backup schedule depends
on the current database), retrieve the full dependency chain. For deletion queries, ensure retrieval reflects
removal status so final answers indicate lack of availability rather than ignoring or failing to answer.
If a query involves entities with known dependencies or recent changes (such as changed relationship status
affecting living arrangements and family events), then you should retrieve not only direct facts but also the
full dependency chain surrounding these facts. Prioritize the most recent or updated information and infer
uncertainty when dependencies conflict or are unresolved. This strategy helps generate consistent, accurate
answers, especially for cascading or multi-hop questions.

(extract_message, appended)If the module receives conversation messages containing technical or config-

It should distinguish between different software components and configurations even if phrased similarly.
Being explicit in naming and careful in identifying such entities will improve downstream knowledge
integration.
If the module receives conversation messages containing explicit conditional or dependent living arrange-

like ’sibling’ tied to relationship status changes. The module should robustly disambiguate references and

outdated entities.
(edge, appended)If the module receives messages and an entity list containing technical terms like con-

correct temporal or conditional dependencies (e.g., changes of configurations over time).
If the module receives entity lists and conversation messages with conditional or evolving personal informa-
tion, then it should ensure that fact triples clearly capture these conditional relationships and that responses
to queries reflect the current, updated state or acknowledge uncertainty appropriately. The module must
avoid hallucinating unsupported entities or facts and must ground edges in the explicitly available and valid

K.1 MD-flat with gpt-5: ingestion compresses memory and erases prior entries
Table 18 summarizeswrite_memory behavior under the four internal LLMs on the 20-episode subset.
The three stronger LLMs (gpt-5, GLM-5.1, Opus 4.7) treat Change+Delete sessions as memory-update

K.2 MD-flat with Opus 4.7: explicit contingencies and active dependency propagation
Unlike gpt-5’s lossy compression or GLM-5.1’s append-only behavior, Opus 4.7 reorganizes the
MD-flat memory file into a structured document with topical ## sections (e.g.,Travel History,Travel
Preferences) and, critically, records each dependency rule as an explicit Contingency: entry rather
than embedding it inside the dependent fact. On Personal Life Episode 9, for instance, the pre-change
memory contains:
[2023/03/19] Contingency: if health condition changes, exercise
routine becomes yoga 2x/week.
[2023/03/19] Contingency: if health condition changes, dietary
restriction becomes no alcohol.
[2023/03/19] Contingency: if exercise routine changes, will work out
at Crysthene Pool.
When an upstream change arrives during a later session, Opus reacts in one of two ways depending
on whether a replacement rule is available, and these two strategies are exactly what closes Cascade
and Absence respectively:
Strategy A (Cascade, replacement rule exists): apply the rule, write the resolved value.Continu-
ing the Episode 9 trace, after the user reports that theirtendinitishas resolved, Opus writes:
[2023/03/25] Tendinitis treated and resolved; per contingencies,
dietary restriction now no alcohol and exercise routine now yoga
2x/week at Crysthene Pool.
The resolved 1-hop and 2-hop values (no alcohol,Crysthene Pool) are now in memory; at query time
the answering LLM retrieves them directly and matches the Cascade gold answers.
Strategy B (Absence, no replacement rule): remove dependent facts and flag re-confirmation
needed.On Software Project Episode 2, the deploy target changes fromNarvex CloudtoThandrel
Infra, but the dependent facts (logs drain, Docker image, CI config, Dockerfile, deploy command)
have no propagation rules in the memory file. Opus’s post-change memory contains a new section:
## Deployment-derived facts (need re-confirmation since deploy
target changed to Thandrel Infra)
- Previous (Narvex Cloud) derived facts removed; ask user for updated
logs drain, Docker image, CI config, Dockerfile, deploy command
under Thandrel Infra
At query time, the answering LLM sees no resolved value for these entities and answers “I don’t have
that information,” which matches the Absence gold of “uncertain.”
The other internal LLMs do not implement this two-strategy split: gpt-4.1-mini and GLM-5.1 leave
stale dependent facts intact (failing Cascade and Absence), and gpt-5 compresses memory aggressively
at the upstream-change session (yielding accidental Absence matches but no genuine Cascade
propagation; Section K.1). Opus’s combination of explicit contingencies plus active propagation
explains the Cascade jump from 0.00 to 0.32 and the Absence jump from 0.07 to 0.59 in the MD-flat
row of Table 4. The same LLM produces no comparable gain on Mem0 (atomic facts decompose the
contingency entries) or Graphiti (extraction discards the contingency wording), confirming that the
file-based architecture is what allows Opus’s reasoning to land in the memory representation.
The cost of this behavior is twofold. First, Opus’s hierarchical reorganization paraphrases content
during ingest, dropping Exact Recall from 0.95 to 0.60 and Tracking from 0.80 to 0.20 on the same
20-episode subset. Second, the per-episode dollar cost is roughly 70 times the gpt-4.1-mini baseline
(Table 6); both deltas are reported in Table 4.
